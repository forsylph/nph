# 재배치 실행계획

## 1. 목적

이 문서는 `03.analysis_results` 재배치 제안안을 실제로 적용할 때의 단계별 실행 계획을 정리한 문서다.

핵심 원칙은 아래와 같다.

- 기존 조사 결과는 잃지 않는다.
- old/보존본을 남긴다.
- 한 번에 다 옮기지 않고, 읽기 경로가 깨지지 않는 순서로 이동한다.
- 실제 이동 전에 항상 문서맵과 기준본을 먼저 만든다.

## 2. 전제

현재 제안 구조:

```text
03.analysis_results/
├─ 030.index/
├─ 031.architecture-framework/
├─ 032.front-miplatform/
├─ 033.runtime-trace/
├─ 034.architecture-emr-viewer/
├─ 035.business-domains/
├─ 036.data-access-infra/
├─ 037.reference-fact/
├─ 038.misc/
└─ 95.추가 검토 사항 및 계획/953.refactoring-ideation/
```

현재 실제 주요 폴더:

- `030.patient-journey`
- `031.Architecture - Framework`
- `032.Architecture - Miplatform`
- `034.Architecture - EMR_viewer`
- `038.Architecture - MISC`
- `95.추가 검토 사항 및 계획/953.refactoring-ideation`

## 3. 실행 원칙

1. `구조 정의 -> 문서맵 -> 기준본 -> 이동` 순으로 간다.
2. 실제 이동 전에는 새 기준본 README/문서셋을 먼저 만든다.
3. 기존 폴더는 바로 삭제하지 않고 보존본이나 old 성격으로 남긴다.
4. trace 문서는 개념 문서와 분리해서 관리한다.
5. fact/todo/reference는 설명 문서와 섞지 않는다.

## 4. 단계별 실행 순서

### 1단계. index 축 확립

목표:
- `030.index`를 `03.analysis_results` 전체의 목차 허브로 만든다.

작업:
- `03.analysis_results/README.md` 유지
- `030.index` 실폴더 신설 또는 기존 `030.patient-journey`와 역할 충돌 여부 정리
- 최소 문서:
  - `README`
  - `문서맵`
  - `용어집`
  - `읽는순서`

주의:
- 현재 `030.patient-journey`가 이미 존재하므로, `030.index`를 새로 열지 아니면 `030.patient-journey`를 `035`로 보내고 `030`을 index로 비울지 먼저 결정해야 한다.

권장:
- 구조상 가장 깔끔한 방식은 `030.patient-journey -> 035.business-domains`로 이동시키고, `030`을 index로 쓰는 것이다.

### 2단계. Front 축 독립

목표:
- `032.Architecture - Miplatform`를 `032.front-miplatform` 관점으로 재정의한다.

작업:
- 폴더명/README/문서맵 보정
- 하위 기준본 생성:
  - `0320.index`
  - `0321.screen-structure`
  - `0322.transaction-mhi`
  - `0323.navigation-command`
  - `0324.dataset-converter`
  - `0325.ui-event-pattern`
  - `0326.jsp-browser-bridge`
  - `0327.fact-check`
  - `0328.todo`
  - `0329.archive`

주의:
- 실제 유지보수 착수점이 Front 쪽이므로, 이 단계는 우선순위가 높다.

### 3단계. Runtime Trace 독립

목표:
- 현재 `031`과 `039-1`에 흩어진 trace 성격 문서를 `033.runtime-trace` 관점으로 모은다.

작업:
- 대표 trace 문서 선별
  - `MD_ORD01001P`
  - `HP_DMS02204M`
  - `EdiMngmPC`
- 템플릿 통일
- trace README/읽는순서 문서 추가

주의:
- 원문은 바로 이동하지 말고, 먼저 복제/기준본으로 모아놓는 것이 안전하다.

### 4단계. Framework와 Data/Infra 분리

목표:
- 현재 `031.Architecture - Framework`의 설명 범위를 더 명확히 나눈다.

작업:
- `031.architecture-framework`
  - 큰 구조, front/data를 포함한 아키텍처 설명 유지
- `036.data-access-infra`
  - `LCommonDao`, `LQueryMaker`, XML Query, TX/Pool, 설정 해설 이동/복제

주의:
- `031`에서 설명 축이 사라지면 안 된다.
- `036`은 상세/심화 쪽으로 두고, `031`은 개요 중심으로 유지하는 편이 안정적이다.

### 5단계. business-domains 정리

목표:
- patient journey, 업무영역별 문서를 `035.business-domains`로 모은다.

작업:
- `030.patient-journey` 문서군 검토
- `az`, `md`, `hp`, `sp` 관점으로 재배치
- 도메인 설명 문서와 runtime trace 문서를 분리

주의:
- patient journey 문서는 구조 설명이라기보다 도메인 흐름 문서이므로 `035`가 더 맞다.

### 6단계. reference-fact 통합

목표:
- `reference`, `fact-check`, `todo`를 느슨하게 한 축으로 관리한다.

작업:
- `037.reference-fact` 아래에
  - reference
  - fact-check
  - todo
  성격을 구획화
- 또는 현재처럼 `031` 내부에서 이미 잘 분리된 구조를 유지하고, 상위에선 링크만 통합

권장:
- 상위 구조에서는 `037.reference-fact`를 두되, 실제 문서 운용은 기존 분리 원칙을 유지하는 편이 안전하다.

### 7단계. EMR/MISC/Refactoring 정리

목표:
- 이미 독립성이 강한 축은 최소 변경으로 정리한다.

작업:
- `034.Architecture - EMR_viewer` -> `034.architecture-emr-viewer`
- `038.Architecture - MISC` -> `038.misc`
- `95.추가 검토 사항 및 계획/953.refactoring-ideation` -> `95.추가 검토 사항 및 계획/953.refactoring-ideation`

주의:
- `039-1`은 이미 목적이 명확하므로 무리한 분해보다 이름/목차 정리가 우선이다.

## 7. 위험과 완화

### 위험 1. 링크 깨짐
- 완화
  - 이동 전 기준본 복제
  - old/보존본 유지
  - 문서맵 우선 정리

### 위험 2. 같은 문서가 여러 축에 중복됨
- 완화
  - 개념 문서와 trace 문서를 역할로 분리
  - 중복이 필요하면 `정본 + 안내 링크`로 처리

### 위험 3. old와 기준본의 충돌
- 완화
  - 기준본 우선 원칙 유지
  - fact-check와 todo를 같이 운용

## 8. 당장 실행할 1순위

지금 바로 실제로 옮기기 시작한다면 가장 먼저 할 일은 이것이다.

1. `030.patient-journey`를 `035.business-domains` 관점으로 보낼지 확정
2. `032.Architecture - Miplatform`를 `032.front-miplatform` 기준본으로 재작성 시작
3. `033.runtime-trace` 실폴더를 만들고 현재 trace 문서를 복제/이관

이 세 가지가 끝나면 전체 재배치의 절반은 닫힌다.
