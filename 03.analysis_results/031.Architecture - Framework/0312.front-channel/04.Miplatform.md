# Miplatform

## 1. 목적

이 문서는 MiPlatform 요청/응답 변환 계층을 현재 코드 기준으로 정리한 기준본이다.

## 2. 기본 흐름

```mermaid
flowchart LR
    DS1[Dataset] --> Req[MiplatformRequest]
    Req --> Conv1[MiplatformConverter]
    Conv1 --> LD[LData/LMultiData]
    LD --> Cmd[Command]
    Cmd --> Conv2[MiplatformConverter]
    Conv2 --> DS2[Response Dataset]
```

## 3. 핵심 해석

- `MiplatformConverter`는 business 계층이 아니다.
- 역할은 MiPlatform Dataset과 DevOn 내부 `LData / LMultiData` 사이의 변환이다.
- 즉 MiPlatform 채널용 adapter 계층이다.

## 4. 직접 확인된 포인트

- `MiplatformServlet`
  - MiPlatform 요청 진입점
- `MiplatformRequest`
  - 요청 Dataset 수신
- `MiplatformConverter`
  - `convertToLMultiDataWithJobType(Dataset ds, Dataset sessionDs)`
  - `convertToDataset(...)`
- `AbstractMiplatformCommand`
  - 변환된 데이터를 command 쪽에서 사용하는 기준점

## 5. 실무 포인트

- Dataset 구조가 기대와 다르면 command보다 converter와 dataset mapping을 먼저 확인한다.
- 저장/수정 계열에서는 `_CUD`와 row 상태가 중요하다.
- MiPlatform 요청은 일반 `.mhi` 추적과 같지만, 앞뒤에 dataset 변환층이 한 번 더 붙는다고 생각하면 된다.

## 6. 현재 문서화 기준

- 기본 프레임워크 생성자는 BIN 계열이 보이더라도, 현재 확인한 NPH 진입 코드에서는 XML 응답 생성 경로가 확인된다.
- 따라서 문서에는 `기본값 BIN`과 `현재 NPH 적용값`을 구분해 적는 것이 맞다.
