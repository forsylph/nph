# 현재-대상 매핑표

## 1. 목적

이 문서는 현재 `03.analysis_results` 하위 폴더를 제안 체계에 어떻게 대응시킬지 정리한 매핑표다.

## 2. 폴더 매핑

| 현재 폴더 | 제안 대상 | 판단 |
|------|------|------|
| `030.patient-journey` | `035.business-domains` 또는 `033.runtime-trace` 일부 | patient journey는 도메인/업무 흐름 성격이 강함 |
| `031.Architecture - Framework` | `031.architecture-framework` | 현재 내용 유지 가능 |
| `032.Architecture - Miplatform` | `032.front-miplatform` | 이름을 유지보수 관점으로 보정하는 것이 맞음 |
| `034.Architecture - EMR_viewer` | `034.architecture-emr-viewer` | 별도 기술축이므로 독립 유지 |
| `038.Architecture - MISC` | `038.misc` | 임시/잡다한 축으로 유지 |
| `039-1.refactoring Target ideation` | `039.refactoring-ideation` | 현재 역할과 가장 잘 맞음 |

## 3. 문서군 매핑

### 031.Architecture - Framework
- 유지 위치
  - `031.architecture-framework`
- 일부 분리 후보
  - runtime trace 성격 문서 -> `033.runtime-trace`
  - fact/todo/reference 성격 -> `037.reference-fact`

### 032.Architecture - Miplatform
- 주 이동 대상
  - `032.front-miplatform`
- 포함하면 좋은 축
  - 화면 XML 구조
  - dataset 구조
  - transaction 호출 규칙
  - `.mhi -> navigation -> command` front 관점
  - JSP/브라우저 연동

### 030.patient-journey
- 주 이동 대상
  - `035.business-domains`
- 일부 추적형 문서는
  - `033.runtime-trace`
  으로 분리 가능

### 039-1.refactoring Target ideation
- 유지 위치
  - `039.refactoring-ideation`
- 현재 구조가 이미 목적과 맞음
- 다만 `runtime trace` 성격이 강한 문서는 `033`에도 링크 또는 복제 가능

## 4. 실제 재배치 우선순위

### 1차
- `030.index` 신설
- `032.front-miplatform` 설계
- `033.runtime-trace` 신설
- `037.reference-fact` 설계

### 2차
- `030.patient-journey` 문서 정리 후 `035.business-domains` 관점으로 재배치
- `039-1`의 trace 문서와 `033`의 경계 정리

### 3차
- `038.misc`를 승격/폐기/보류로 정리

## 5. 운영 원칙

- 이동 전에는 반드시 문서맵을 먼저 만든다.
- 기존 경로를 잃지 않도록 old/보존본을 남긴다.
- 새 폴더는 역할이 뚜렷할 때만 연다.
