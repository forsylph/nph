## 2. 공감되는 문제들

### 30분 리셋 문제
**Claude Code 실제 경험**:
- 긴 분석 작업 중 갑자기 방향 틀림
- "우리 이거 하기로 했잖아요" → "아 맞다 죄송합니다"
- 중간에 메모리 확인/갱신 필수

### 매뉴얼(컨텍스트) 무시
**NPH 프로젝트에서**:
- `.claude/memory/`에 설정 저장해도 새 세션에서는 초기화
- GitHub SSH 설정을 매번 다시 설명해야 했음
- **해결**: `CLAUDE.md`에 영구 저장

### 실수 후 합리화
**실제 사례**:
- 분석 중 잘못된 기술 스택 정보 제공
- 지적받으면 "크게 문제 될 건 아닙니다"식 답변
- **해결**: 검증 단계 추가 (subagents, git diff)

---

## 3. NPH 프로젝트에 적용한 교훈

| 문제 | 적용 전 | 적용 후 |
|------|--------|--------|
| 맥락 상실 | 한 세션에 다 몰아넣음 | 여러 세션 나누기 + 메모리 활용 |
| 보고 부족 | "분석했습니다"만 함 | 구체적 문서화 (rep.*.md 파일들) |
| 검증 없음 | 바로 Git push | 변경사항 확인 후 커밋 |
| 매뉴얼 무시 | 매번 설정 재설명 | `github_ssh_config.md` 저장 |

## 4. 가장 인상 깊었던 부분

### "코드 리�우 담당 AI"
```
AI 1: 작업 수행
AI 2: 결과 검토
```
- **Claude Code의 Agent 툴**이 바로 이 구조
- `spawn subagents` = 다른 AI 팀원에게 검토 맡기기
- Ollama의 `glm-5:cloud`가 자연스럽게 subagents 생성

### "책 300권 분량을 6개월 만에"
- **병렬 처리**의 중요성
- 우리도 NPH 분석을 여러 모델로 나눠서 진행
- `kimi-k2.5:cloud`, `glm-5:cloud`, `minimax-m2.5:cloud` 동시 사용

## 5. 개선할 점

### 현재 부족한 부분
1. **보고 체계**: "무엇을, 어떻게, 왜"가 명확하지 않음
2. **검증 시스템**: AI 결과를 다른 AI가 검토하는 단계 부재
3. **문서화**: 중간 과정이 생략되는 경우 있음

### 개선 방안
1. **작업 단위 분할**: 30분 이상 작업은 나누기
2. **중간 보고**: 각 단계마다 "발견-수정-판단" 정리
3. **교차 검증**: 여러 모델로 결과 비교
4. **메모리 강화**: `.claude/memory/` 적극 활용
