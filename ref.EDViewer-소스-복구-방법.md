# EDViewer 소스 코드 복구 방법

> 자체 개발했으나 소스 코드를 분실한 경우의 복구 전략

---

## 상황 파악

### 가능한 시나리오

| 시나리오 | 설명 |
|---------|------|
| **개발자 퇴사** | 개발자가 소스를 가져감 |
| **저장소 손상** | SVN/Git 서버 장애 |
| **외주 개발** | 하청업체가 소스 미전달 |
| **백업 분실** | 백업 미관리로 소스 유실 |
| **노트북 분실** | 개발용 PC 분실/도난 |

## 복구 방법 (합법적)

### 방법 1: 개발자/업체 연락

```
1차: 원 개발자 연락
├── 개인 PC에 소스가 있을 가능성
├── 개인 GitHub/Bitbucket 저장소
└── USB/외장하드 백업본

2차: 하청업체 연락 (외주 개발 시)
├── 계약상 소스 인계 확인
├── 업체 내부 저장소 확인
└── 법적 조치 고려

3차: 동료 개발자 확인
├── 공유 폴더/메일 첨부
├── 코드 리뷰 기록
└── 배포 패키지 내부 (소스맵 등)
```

### 방법 2: 백업 복구

```bash
# 1. 서버 백업 확인
/mnt/backup/
├── svn/                    # SVN 백업
├── git/                    # Git 백업
└── source/                 # 소스 백업

# 2. 개발자 PC 밁업
C:\Users\[개발자]\workspace\
C:\Users\[개발자]\Documents\

# 3. 메일 첨부 파일 검색
*.zip, *.tar.gz, *.jar
"EDViewer", "eView", "EMR"

# 4. NAS/공유 폴�드
\\server\shared\project\
```

### 방법 3: IDE 설정 파일 복구

```
이클립스 워크스페이스 복구
├── .metadata/            # 프로젝트 목록
├── RemoteSystemsTempFiles/
└── Servers/

복구 가능한 것:
├── 프로젝트 구조
├── 빌드 설정
└── 일부 소스 (자동 저장된)
```

### 방법 4: 디버깅 심볼에서 복구

```
PDB/Debug Symbol 파일
├── EDViewer.pdb          # Visual Studio 디버그 심볼
├── .map 파일              # 맵 파일
└── .dbg 파일              # 디버그 정보

복구 가능한 것:
├── 함수명
├── 변수명
├── 소스 파일 경로
└── 코드 라인 정보
```

### 방법 5: 기존 개발 문서

```
문서에서 소스 복원
├── 설계 문서
│   └── 클래스 다이어그램
├── API 문서
│   └── 인터페이스 정의
├── 시퀀스 다이어그램
└── 주석이 포함된 바이너리
```

## 역공학 없이 복원 가능한 것

### 가능한 복원

| 항목 | 방법 | 완성도 |
|-----|------|--------|
| **인터페이스 정의** | OLE/COM Viewer | 100% |
| **리소스** | 리소스 해커 | 100% |
| **문자열** | strings 명령 | 100% |
| **API 목록** | Dependency Walker | 100% |
| **프로토콜** | 네트워크 분석 | 80% |

### 불가능한 복원

| 항목 | 이유 |
|-----|------|
| **로직** | 기계어 → 고수준 언어 불가 |
| **알고리즘** | 최적화된 코드 역분석 어려움 |
| **주석** | 컴파일 시 제거됨 |
| **변수명** | 심볼 없으면 복원 불가 |

## 실제 복구 시도

### 1. COM 인터페이스 복구

```bash
# OLE/COM Object Viewer로 인터페이스 추출
$ oleview EDViewer_Ocx.ocx

# 출력: IDL 파일
[
    uuid(xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx),
    version(1.0)
]
interface IEDViewer : IDispatch
{
    [id(1)] HRESULT LoadDocument([in] BSTR url);
    [id(2)] HRESULT Show();
    // ... 메소드 목록
}
```

**이것만 복구 가능** - 실제 구현 내용은 없음

### 2. 리소스 추출

```bash
# 아이콘, 이미지, 문자열 추출
$ resource_hacker EDViewer_Ocx.ocx -extract

# 추출 가능한 것:
├── Icon/Bitmap
├── Dialog template
├── String table
└── Version info
```

### 3. 문자열 분석

```bash
$ strings EDViewer_Ocx.ocx | grep -i error

# 오류 메시지 확인 가능
"Failed to load document"
"Invalid parameter"
"Out of memory"
```

## 현실적인 권고

### 짧은 답: 소스 복구는 거의 불가능

```
상황: 자체 개발 + 소스 분실

결론:
├── 완전한 소스 복구: 불가능
├── 부분적 복구 (인터페이스): 가능
└── 재개발: 더 현실적

비교:
├── 복구 시도: 1~2년, 불확실한 결과
└── 재개발: 6~12개월, 확실한 결과
```

### 권장하는 접근

```
1단계: 최선의 노력으로 복구 시도 (2주)
   ├── 모든 백업 확인
   ├── 개발자/업체 연락
   └── 문서 정리

2단계: 복구 실패 시 재개발 결정
   ├── 요구사항 정의
   ├── 기술 스택 선정
   └── 개발 계획 수립

3단계: 점진적 마이그레이션
   ├── 새 뷰어 개발
   ├── 병행 운영
   └── 완전 전환
```

## 결론

### 핵심 메시지

> **자체 개발한 EDViewer라도 소스가 분실되면 완전한 복구는 거의 불가능합니다.**

**이유:**
- C++ 네이티브 바이너리는 완전한 역컴파일 불가
- 7.7MB 규모의 로직 복원은 불현실적
- 시간/비용이 재개발보다 큼

**권장:**
1. 모든 가능한 복구 방법 시도 (2주)
2. 실패 시 재개발 또는 상용 제품 도입 검토
3. 향후 소스 관리 체계 확립
